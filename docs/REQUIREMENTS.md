# 需求文档

## 1. 项目背景

长篇网络小说创作不是单次文本生成，而是一个长期叙事系统。作者需要持续管理主线、人物、伏笔、节奏、读者期待、世界状态和临时灵感。

现有 AI 写作方式通常有两个问题：

1. 通用聊天模型依赖对话上下文，长篇稳定性不足。
2. 固定流程的写作工具缺少 agent 式自我规划、回看、复盘和变更能力。

本项目希望构建一套面向 agent 的长篇小说创作系统，让 AI 能基于文件化记忆自主启动、规划、写作、审查和更新小说项目。

## 2. 目标用户

MVP 阶段的目标用户是项目作者本人，或者愿意使用 agent 和文件系统进行创作的高级用户。

未来潜在用户包括：

- 长篇网文作者
- AI 写作实验者
- 小说工作室
- 互动叙事和游戏剧情创作者
- 想从一个创意种子自动孵化小说的普通用户

## 3. 产品定位

本项目不是普通 AI 写作助手。

普通写作助手的模式是：

```text
用户给大纲、人物和章纲，AI 帮忙写正文。
```

本项目的目标模式是：

```text
用户给创作种子，agent 建立作品记忆、规划故事、创建人物、滚动写作、维护状态，并在用户介入时进行变更管理。
```

## 4. 核心用户故事

### 4.1 新书启动

作为用户，我可能只有一个设定、一个主角形象、一个爽点或一句故事梗概。

我希望 agent 能：

- 分析这个 seed 的长篇潜力。
- 给出多个创作方向。
- 建立创作宪法。
- 创建初始世界观、人物和第一卷目标。
- 生成近期 9-15 章滚动大纲。
- 建立第一条可跨轮延展的连续剧情流。
- 初始化项目记忆文件。

### 4.2 连续写作

作为用户，我希望 agent 能每轮写 3 章，并且每章都：

- 编译上下文。
- 理解前文。
- 规划本章。
- 写正文。
- 自我审稿。
- 更新记忆。
- 捕捉灵感。

### 4.3 中途变更

作为用户，我可能在写到一半时想到新点子，例如新反转、新人物关系、新世界真相。

我希望 agent 不要直接重启整本书，而是：

- 判断变更影响级别。
- 评估是否符合创作宪法和读者承诺。
- 给出接入方案。
- 更新灵感池、伏笔、规划或相关状态。

### 4.4 冷启动接续

作为用户，我希望一个新会话中的 agent 只依靠项目文件，就能理解小说当前状态并继续写。

这意味着项目文件必须是唯一可信长期记忆。

### 4.5 审查和复盘

作为用户，我希望 agent 能检查小说是否正在跑偏，包括：

- 叙事债是否拖太久。
- 人物行为是否降智。
- 信息可见性是否穿帮。
- 世界状态是否持续反应。
- 近期章节是否水。
- 是否符合目标读者期待。

## 5. 功能需求

### 5.1 小说启动

- 输入创作 seed。
- 生成创作方向。
- 建立创作宪法。
- 初始化全书记忆。
- 初始化第一卷。
- 初始化人物、势力、地点、物品。
- 初始化动态账本。
- 生成近期规划。

### 5.2 记忆系统

系统必须支持以下记忆层：

- 全书层
- 卷层
- 章群层
- 章节层
- 实体层
- 动态账本层

动态账本必须至少包含：

- 叙事债务
- 伏笔
- 信息可见性
- 世界状态
- 灵感池
- 角色意图

### 5.3 上下文编译

写作前必须生成上下文包。

上下文包必须落盘为标准化 `context_pack.md`，用于保证实验可复现和冷启动可审查。

上下文包应遵循：

```text
远处读摘要
近处读原文
关键旧内容按需回看
未来章纲每轮全文读取
动态账本按本章涉及对象定向读取
context pack 只沉淀工作记忆，不复制项目资料库
```

### 5.4 三章一轮生产批次

每轮默认写 3 章，但三章只是生产批次，不是叙事单位。章节必须处在 active_flow 连续剧情流中，从上一章外部交接承接，并把具体压力交给下一章。

每章必须独立完成：

- 理解
- 规划
- 写作
- 审查
- 状态更新

同一轮可以共享会话，但不能共享未落盘的隐性记忆。

### 5.5 状态更新

每章写完后必须更新：

- 章节摘要
- canon delta
- 人物状态
- 世界状态
- 信息可见性
- 叙事债务
- 伏笔表
- 灵感池
- 近期规划
- active_flow 连续剧情流

### 5.6 变更管理

系统必须支持用户中途修改想法。

变更级别：

```text
Level 1: 风格变更
Level 2: 局部设定变更
Level 3: 人物/关系变更
Level 4: 主线/卷纲变更
Level 5: 作品 DNA 变更
```

只有 Level 5 或用户明确要求重启时，才考虑重新进入启动流程。

### 5.7 审查

审查必须覆盖：

- 节奏
- 跨章连续性和 round 边界是否被误写成叙事边界
- 爽点
- 人物意图
- 信息可见性
- 世界反应
- 叙事债务
- 伏笔
- 风格一致性
- 记忆完整性

## 6. 非功能需求

### 6.1 可读性

项目文件必须能被人类直接阅读和修改。

### 6.2 可迁移性

文件结构不绑定单一 agent 或模型。

### 6.3 可恢复性

新会话必须能通过读取项目文件恢复创作状态。

### 6.4 可演进性

MVP 的文件结构未来应能迁移到数据库和 UI 产品。

### 6.5 可控性

agent 不能静默修改创作宪法、主角核心动机、终局级秘密等重大内容。

受保护文件必须有修改前确认、diff 摘要和 checkpoint。

## 7. 成功标准

MVP 成功的判断标准：

- 只给一个 seed，系统能初始化一本可写的小说。
- 连续写到 15-30 章后，人物、伏笔、世界状态仍然稳定。
- 新会话读取项目文件后，可以准确接续写作。
- 用户中途加入新点子，系统能评估并接入，而不是直接跑偏。
- 叙事债务能被记录、推进和偿还。
- 角色有自己的目标，不只是服务主角。
- 世界能对主角行动持续反应。
